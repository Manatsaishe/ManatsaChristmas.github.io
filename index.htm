INCIDENT LOG · LAB ENVIRONMENT

Linux Privilege Escalation via SUID Binary

Analyzing a compromised Linux host to identify and exploit a misconfigured SUID binary, resulting in root-level access.

READ FULL LOG >>

1. Initial Discovery

During enumeration of the target box after gaining an initial low-privilege shell (`www-data`), I ran a standard search for binaries with the SUID bit set.

find / -perm -u=s -type f 2>/dev/null

2. Vulnerability Identification

The output revealed a non-standard binary `/usr/bin/system-check` running with SUID root permissions. Analysis using `strings` and basic execution showed it was vulnerable to path injection.

www-data@target:/tmp$ ls -la /usr/bin/system-check -rwsr-xr-x 1 root root 16520 Feb 28 14:22 /usr/bin/system-check www-data@target:/tmp$ strings /usr/bin/system-check | grep "service" service apache2 status

3. Exploitation & Root Path Graph

I created a malicious script named `service` in `/tmp` that spawned `/bin/bash`, made it executable, and manipulated the `PATH` variable before running the vulnerable SUID binary.

Fig 1. Attack Path: User -> Malicious 'service' script -> Root Shell

4. Outcome

The SUID binary executed my malicious `service` script with root privileges, dropping me into a root shell.

root@target:~# whoami
root

Analyzing Malicious Docm: "Invoice_Q4.docm"

Reverse engineering a suspicious macro-enabled Word document designed to establish C2 communication and drop a second-stage payload.

READ FULL LOG >>

1. Executive Summary

The file "Invoice_Q4.docm" is a malicious Word document utilizing VBA macros to execute obfuscated PowerShell commands. It attempts to download a payload from a remote C2 server. The file is classified as a dropper for an Emotet-style botnet.

2. Static Analysis & IOCs

File Type: Microsoft Word 2007+ Document (Macro-Enabled)

3. Dynamic Analysis (Sandbox Execution)

Upon opening the document and enabling macros in a sandbox environment, the following process execution chain was observed:

INFRASTRUCTURE PROJECT · AZURE AD

Implementing Conditional Access & Intune Compliance

Project log detailing the hardening of an M365 tenant using Conditional Access policies to enforce MFA and require compliant devices.

READ FULL LOG >>

1. Project Scope

The goal was to move from implicit trust to a Zero Trust architecture. We required all users to pass MFA and ensure their devices were enrolled and compliant in Microsoft Intune before accessing Exchange Online or SharePoint.

2. Implementation: Device Compliance Policy

We created a Windows 10/11 Compliance Policy in Intune requiring BitLocker encryption, Secure Boot, and minimum OS version. Devices failing these checks are marked "non-compliant."

Microsoft Endpoint Manager admin center
Policy Name: Win10_Secure_Baseline

Require BitLocker
Require Secure Boot to be enabled
Minimum OS: 10.0.19041

3. Implementation: Conditional Access Flow

A Conditional Access policy was configured to block access unless the "Require MFA" and "Require device to be marked as compliant" controls were satisfied.
